# Security Remediation — June 2026

This branch (`fix/security-hardening-2026-06`) removes a privilege-escalation
backdoor and applies a batch of security hardening. Some items require **manual
action by the repository owner** that code alone cannot perform. Work through
the checklist below.

---

## 1. Rotate the exposed admin credential (required)

The previous `ensure_admin` management command hardcoded an admin password
(`Thisisourteamproject`) in source. It is present in **6 historical commits**
(introduced in `932af19`), so it must be treated as **public and compromised**,
even though this branch removes it from the current tree.

**Action:**
1. Log in to the production Django admin (`/admin/`) and change the password of
   the admin account, **or** run the new env-driven bootstrap (see §2) with a
   fresh password.
2. Pick a strong, unique password (not reused anywhere). The validators require
   ≥10 chars, mixed case, a digit, and a symbol.
3. If that password was reused for any other service, rotate it there too.

> Scope note: this password is the **only literal secret** that was committed.
> `DJANGO_SECRET_KEY` uses `generateValue: true` in `render.yaml`, so the
> production secret key was generated by Render and was **never** the value in
> the local (untracked) `.env`. `FIELD_ENCRYPTION_KEY`, `GEMINI_API_KEY`,
> `DATABASE_URL`, `REDIS_URL`, etc. are Render `sync:false` vars / untracked
> `.env` entries and are **not in git** — rotate them only if you suspect
> exposure through another channel. (Do **not** rotate `FIELD_ENCRYPTION_KEY` —
> it would orphan all existing encrypted `pii_data`.)

---

## 2. Set the new admin bootstrap env vars (optional, recommended)

`ensure_admin` now reads credentials from the environment and no-ops if they are
absent. To use it, set in the Render dashboard (do **not** put these in
`render.yaml`):

```
DJANGO_SUPERUSER_USERNAME=<admin username>
DJANGO_SUPERUSER_EMAIL=<admin email>
DJANGO_SUPERUSER_PASSWORD=<the freshly rotated password>
```

Then run once (Render shell): `python manage.py ensure_admin`. The command is
idempotent and safe to re-run.

---

## 3. Other recommended environment variables

| Variable | Why | Default if unset |
|---|---|---|
| `JWT_SIGNING_KEY` | Lets the JWT key rotate independently of `DJANGO_SECRET_KEY`. | Falls back to `DJANGO_SECRET_KEY` (unchanged behaviour). |
| `REDIS_URL` | Shared cache for throttling/lockout/OTP across workers. | Falls back to the **database** cache (now shared, correct, but slower). |
| `TRUSTED_PROXY_COUNT` | Number of trusted proxies in front of the app for safe client-IP parsing. | `1` (correct for Render). |

## 4. Frontend follow-up (required for the profile page)

Changing a profile email now requires re-authentication: the `PUT/PATCH
/api/profile/` request must include `current_password` when `email` changes, and
email changes are rejected for OAuth accounts. **Update the profile form** to
prompt for the current password before submitting an email change, or that
field will return `400 "Current password is required to change email."`

---

## 5. Deferred hardening (tracked for a follow-up PR)

- **Refresh tokens in `localStorage` (L7).** Tokens are exfiltratable by any XSS
  (mitigated today by the strict `script-src 'self'` CSP). Moving the refresh
  token to an `httpOnly`, `Secure`, `SameSite` cookie needs a coordinated
  backend (set-cookie + refresh endpoint) and frontend (`AuthContext`) change;
  it was kept out of this PR to avoid rushing an auth refactor onto a
  prod-auto-deploy branch.
- **DB-level unique email constraint (M3).** Enforced in app code here. A
  database `UNIQUE` (partial, `WHERE email <> ''`) index is stronger but must be
  preceded by a production data audit to de-duplicate any existing rows, or the
  migration will fail mid-deploy.
- **CSP `style-src 'unsafe-inline'`.** Tighten with hashes/nonces once inline
  styles are inventoried.

## 6. Purge the credential from git history (optional but recommended)

Removing the secret from the working tree does not remove it from history.
Anyone can still `git log -p` and recover `Thisisourteamproject`. Because the
password is being rotated (§1), history rewrite is **defence-in-depth**, not
strictly required — but recommended for a public repo.

> ⚠️ **This rewrites history.** It changes commit hashes, requires a
> force-push, breaks every open PR/branch (including the `dependabot/*`
> branches), and forces all collaborators to re-clone. Coordinate first.
> Do it when no other PRs are mid-flight, and re-open Dependabot afterwards.

### Recommended: BFG (surgical — redacts the secret, keeps file layout)

```bash
# 1. Mirror-clone a fresh copy
git clone --mirror https://github.com/yokesh-kumar-M/Piicasso.git piicasso-mirror.git
cd piicasso-mirror.git

# 2. Redact the literal secret everywhere it appears in history
printf 'Thisisourteamproject==>REDACTED\n' > ../replacements.txt
bfg --replace-text ../replacements.txt

# 3. Clean up and force-push the rewritten history
git reflog expire --expire=now --all && git gc --prune=now --aggressive
git push --force
```

### Alternative: git-filter-repo (also redacts the string)

```bash
pip install git-filter-repo
git clone https://github.com/yokesh-kumar-M/Piicasso.git
cd Piicasso
printf 'Thisisourteamproject==>REDACTED\n' > ../replacements.txt
git filter-repo --replace-text ../replacements.txt
git remote add origin https://github.com/yokesh-kumar-M/Piicasso.git
git push --force --all && git push --force --tags
```

### After the rewrite
1. Tell every collaborator to delete their local clone and re-clone.
2. Re-trigger Dependabot (close/reopen its PRs or bump the config) since its
   branches will be orphaned.
3. Rotate the password (§1) regardless — assume the old one was harvested while
   it was public.
4. Consider enabling GitHub **secret scanning / push protection** so a future
   committed secret is blocked at push time.
